In React applications, state is needed whenever something needs to change due to an interaction.

### What is State?

`State` literally means condition or status. Different states have different conditions.

For example, when you click a `Button`, a `Modal` will open. The open state of the `Modal` is one condition, and when the `Modal` is closed, it's another condition. This is what we call `State`.

Since React helps with reactive applications, React needs to understand when something has changed in the application that requires re-rendering, and it needs to remember what condition was in what state. This is why it needs to manage various states of the application. React does this through its built-in `hook` called `useState`.

### Can't we do these tasks conditionally with regular variables?

No, we can't do that in React applications because when something changes in a React application, its component needs to be re-rendered and the `UI` needs to be updated. But if we change the value in a local variable, React can't remember it and doesn't trigger a re-render. You might think, "We could just manually update the `UI`," and yes - but then why are we using React? We're using React so that when something changes, the `UI` updates automatically.

### What doesn't happen with local variables:

1. **Local variables don't persist between renders:** This means that when a component re-renders, the entire component renders again from scratch. It doesn't remember what changes were made to local variables in the previous render.

2. **Changes to local variables won't trigger renders:** This means that when a local variable changes, React doesn't even know that it needs to re-render the component.

---

**To update a component with new data, two things are needed:**

1. **Retain the data between renders:** This means that after re-rendering, the component needs to know what changed in its previous render, which it will update in the `UI`.

2. **Trigger a Re-render:** This means that when any state changes, React needs to trigger its re-render.

---

**The `useState` hook provides these two things:**

1. **A State Variable:** The `useState` hook provides a state variable that it can remember across each re-render.

2. **A Setter Function:** The `useState` hook provides a `setter function` that can change the state variable.

### `useState` hook - how to write it and how to use it?

**Syntax:**

```jsx
const [currentValue, setterFunction] = useState(initialValue);
```

```jsx filename="App.jsx" showLineNumbers {4}
import { useState } from "react";

export default function App() {
const [count, setCount] = useState(0);

    return (
    <>
         <h1>Count:  </h1>
        <button>Click to Increment Count</<button>
    </>
    );
}
```

### Change the value of the `State variable` using the `setter function` and use the value.

```jsx filename="App.jsx" showLineNumbers {7,11-12}
import { useState } from "react";

export default function App() {
const [count, setCount] = useState(0);

    function handleCountIncrement() {
        setCount(count + 1);
    }
    return (
    <>
         <h1>Count: {count}  </h1>
        <button onClick={handleCountIncrement}>Click to Increment Count</<button>
    </>
    );
}
```

**In React, all functions that start with `use` are hooks. Not just `useState`, React has many other built-in hooks that we will learn about later. Each of these hooks adds different features to React. So React hooks are special functions. Therefore, they should not be treated like normal functions. And React hooks must always be used at the top level of a component. They should never be used inside a condition, loop, or other function.**

### How does React's `useState` hook work?

When we write `useState` inside a component, it means telling React that this component has some state that it needs to remember.

```jsx
const [index, setIndex] = useState(0);
```

In this case, we have taken a state variable `index`. So React will remember the value of `index` in its memory for each render.

<Callout
    type='info'
    emoji='📔'>
    The naming convention for `State` is - whatever name is given to the state variable,
    the `setterFunction` should also follow that name. For example: `const
    [index, setIndex] = useState(0);`
</Callout>

#### Render and re-rendering process:

```jsx
const [index, setIndex] = useState(0);
```

![State](../../../../public/state.svg)

1. **When the component renders for the first time:** It returns the initial value we provided for the state variable. In this case, `initial value = 0`, so the state will return `0`.

2. **When the value of the state variable is updated:** When the value is updated using the `setter function` due to a user click or some interaction `setIndex(index + 1)`, the state value changes to `1` and tells React to remember it, and then the `setter function` calls React's `Render()` method to trigger another render.

3. **When it renders for the second time:** The component renders again from scratch, and React sees that `useState(0)` has an `initial value 0`, but when the `state value` changed in the previous render, React remembered it, so it will return the value of `index` as `1`.

And this is how each render will continue.

### What if a component needs multiple state variables?

If a component needs to manage multiple states, multiple `useState` calls can be made. There's no problem with that. However, if one state is related to another, we can also store state values as an `Array` or `object`.

### State is a component's private data

React state is confined to its component. State is called a component's private data.

Let's say a state is used in a component. Now, no matter how many times that component is used, the state for each will be separate. It's not that if one's state changes because of component reuse, the others will also change their state. This will never happen. Because each component's state remains in an isolated condition within it and they are unique.

However, if our requirement is such that the state of each reusable component will remain `sync` or if changing the state in one will affect each one, then we can use the `lifting state up` technique. We will learn about this in detail later.
